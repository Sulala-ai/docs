## Overview

Skills are **documentation-only** Markdown plus optional **YAML tools**. The agent loads one **folder per skill** containing instructions (README) and, if needed, custom tools defined in `tools.yaml`. There is no per-skill code in the agent—only generic tools plus any tools you define in YAML.

## Skill folder layout

Each skill is a **folder** with:

| File | Required | Description |
|------|----------|-------------|
| **README.md** or **SKILL.md** | Yes | Markdown with YAML frontmatter (`name`, `description`) and the skill body (when/how to use). This is what the model sees as context. |
| **tools.yaml** | No | Optional. Defines custom tools for this skill (e.g. API calls). The loader expects a `tools:` array; each entry is one tool. If you don’t need custom tools, you can omit this file or use `tools: []`. |

Example layout:

```
~/.sulala/workspace/skills/my-skill/
├── README.md      # Instructions (frontmatter + body)
└── tools.yaml     # Optional: custom tools (tools: [...] or tools: [])
```

The store and hub use the same layout: `data/skills/<slug>/README.md` and optional `data/skills/<slug>/tools.yaml`.

## Create your first skill (quick)

**Option A — Ask the AI chat**  
In the dashboard chat, ask the agent to create a skill. For example: *"Create a hello world skill"* or *"Create an example skill called helloworld that says hello to the user."* The agent will create the folder with `README.md` and an empty `tools.yaml` in your workspace skills directory. Refresh skills or restart the gateway, then try it in chat (e.g. “Say hello”).

**Option B — Create manually**

1. **Create a folder** for the skill (use a name that matches the skill’s purpose, e.g. `hello-world`):
   ```bash
   mkdir -p ~/.sulala/workspace/skills/hello-world
   ```

2. **Add the skill file** — `README.md` or `SKILL.md` in that folder. Minimal content:

   ```markdown
   ---
   name: hello-world
   description: Use when the user says hello or asks for a greeting.
   ---

   # Hello World

   Use when the user wants a greeting or says hello.

   ## How to respond

   Reply with a short, friendly greeting. You can use the user's name if they shared it.
   ```

3. **(Optional) Add tools** — If the skill needs custom API tools, add `tools.yaml` in the same folder with a `tools:` array (see [Adding tools (tools.yaml)](#adding-tools-toolsyaml) below). Otherwise you can add an empty `tools.yaml` with `tools: []` for later, or omit it.

4. **Load the skill** — Restart the agent gateway, or wait for the skills watcher (`SKILLS_WATCH=1`, default) to pick it up. In the dashboard, the skill should appear in the skills list.

5. **Try it** — In chat, ask something that matches the description (e.g. “Say hello”). The agent will use this skill’s instructions (and any tools from `tools.yaml`) to answer.

For more options (optional frontmatter, full tools schema, publishing to the store), see the sections below.

## Skill format (reference)

Skills are `.md` files with YAML frontmatter. At minimum:

```yaml
---
name: my-skill
description: One-line summary of when to use this skill.
---
```

### Required frontmatter

- **name** — Unique identifier; used for precedence, config, and registry.
- **description** — Shown in the "Available skills" list so the agent knows when to apply the skill.

### Optional frontmatter

- **metadata** — JSON object; e.g. `sulala.requires.bins` for load-time gating (skill only loaded if those binaries exist):

```yaml
---
name: git
description: Git operations
metadata:
  {
    "sulala": {
      "requires": { "bins": ["git"] }
    }
  }
---
```

- **homepage** — URL for docs (optional).
- **version** — Semver string for registry (optional).

## Skill body

The body is Markdown. It should explain:

1. **When to use** — Match the description; clarify edge cases.
2. **How to use** — Which commands to run (binary + args). The agent uses generic tools; the skill describes which `run_command` / `read_file` / `run_task` to call.
3. **Limits** — What the agent must not do; tools that need `ALLOWED_BINARIES` or other safety settings.

### Example structure

```markdown
# Skill Name

Use when the user asks for X, Y, or Z.

## Setup

- Required binaries: `foo`, `bar` (add to ALLOWED_BINARIES in agent config).
- Required env: `FOO_API_KEY` (user sets in skill config or .env).

## Commands

- **List:** Use `run_command` with `binary: "foo"`, `args: ["list"]`.
- **Create:** Use `run_command` with `binary: "foo"`, `args: ["create", "--name", "X"]`.

## Limits

- Do not run destructive commands without explicit user confirmation.
```

## Generic tools the agent has

- **run_task** — Enqueue a background task (type + payload).
- **read_file** — Read a file under `AGENT_WORKSPACE_ROOT` (if set).
- **write_file** — Write a file under the workspace (when enabled).
- **run_command** — Run a single binary with args. Skills document which commands to run; the agent follows the doc and calls this tool. Restrict allowed binaries via `ALLOWED_BINARIES` in config.
- **run_agent** — Call another session (sub-agent), one level only.

For **custom API tools** (HTTP calls, integrations), add a **tools.yaml** in the skill folder instead of writing code. See below.

## Adding tools (tools.yaml)

You can add **custom tools** to a skill by placing a **tools.yaml** file in the same folder as the skill’s README. The agent loads every skill folder that contains `tools.yaml` and registers one tool per entry in the `tools:` array. No TypeScript needed.

### Where to put it

- **Local skill:** Same folder as the skill’s README, e.g. `~/.sulala/workspace/skills/my-skill/tools.yaml`.
- **Store / hub:** `data/skills/<slug>/tools.yaml`. When users install the skill, the agent fetches both README and tools.yaml.

### YAML format

The file must have a top-level **`tools:`** array. Each item is one tool. Minimal example (no custom tools yet):

```yaml
# Add tool definitions for this skill. Schema: name, description, profile, auth, request, parameters, response.
tools: []
```

Example with one GET tool:

```yaml
tools:
  - name: my_list_items
    description: "List items from the API. Use when the user asks to list or show items."
    profile: full          # full | messaging | coding | minimal
    auth: none            # none | stripe_secret_key | discord_bot_token | portal
    request:
      method: GET
      url: https://api.example.com/v1/items
      queryParams: { limit: limit }
    parameters:
      - name: limit
        type: number
        description: Max items to return (default 10)
        default: 10
    response:
      listPath: data       # JSON key that holds the array
      itemKeys: [id, name]
      outputKey: items
      countKey: count
```

### Tool fields (reference)

| Field | Description |
|-------|-------------|
| **name** | Unique tool name (e.g. `stripe_list_customers`). Must be unique across all skills. |
| **description** | One-line description for the model (when to call this tool). |
| **profile** | `full` \| `messaging` \| `coding` \| `minimal` — which agent profiles can use this tool. |
| **auth** | `none` \| `stripe_secret_key` \| `discord_bot_token` \| `portal` — how the request is authenticated. `portal` uses Integrations (OAuth) via `PORTAL_GATEWAY_URL` + `PORTAL_API_KEY`. |
| **request** | Single HTTP call: `method`, `url`, `queryParams`, and for POST: `bodyType` (`json` or `form`), `bodyKeys` or `body`. |
| **steps** | Multi-step flow (multiple HTTP calls). Each step can use `{{step0.id}}` etc. from previous responses. Use instead of `request` when the tool needs more than one call. |
| **parameters** | Array of `{ name, type, description, default? }` for the model. |
| **response** | How to shape the JSON: `listPath` or `rootIsArray`, `itemKeys`, `outputKey`, `countKey` for lists; `singleKeys`, `outputKey` for single objects. |

- **URL and params:** Use `{{arg_name}}` in `url` for path params. `queryParams` is a map from API param name to arg name.
- **POST body:** `bodyType: json` with `bodyKeys: [arg1, arg2]`, or `bodyType: form` with `body: { param: arg_name }`.

For **multi-step flows** (e.g. create resource then call another endpoint), use **`steps`** instead of **`request`**. Each step can reference previous step output via `{{step0.id}}`, `{{step0.field}}`, etc. See the agent repo `agent/docs/tool-spec-format.md` for the full schema and examples.

## Where skills load from (precedence)

1. **User** — `~/.sulala/workspace/skills/<name>/README.md` or `SKILL.md` (safe from project updates)
2. Workspace — `AGENT_CONTEXT_PATH`
3. Managed — `~/.sulala/skills` (flat `.md` files)
4. Bundled — `context/` in the agent
5. Extra — `SKILLS_EXTRA_DIRS`

On name conflict, the higher source wins. To keep a skill from being overwritten on updates, use `~/.sulala/workspace/skills/<slug>/README.md`.

## Adding to the Store (publish so others can install)

1. **Store repo:** Add skill content under `store/data/skills/<slug>/README.md` (and optionally `store/data/skills/<slug>/tools.yaml`). Legacy flat `data/skills/<slug>.md` is still supported for README-only.
2. **Registry:** Add an entry to `store/data/registry.json`:

```json
{
  "skills": [
    {
      "slug": "my-skill",
      "name": "my-skill",
      "description": "One-line description.",
      "version": "1.0.0"
    }
  ]
}
```

The store serves content from `data/skills/<slug>/README.md` and injects the `url` in the registry response. Users set `SKILLS_REGISTRY_URL` to the store and run `sulala skill install my-skill`.

**Publish from the agent:** In the dashboard, use **Publish to store** (e.g. from My skills). The agent sends the skill to the store’s submission API. Store URL is derived from `SKILLS_REGISTRY_URL`. Set `STORE_PUBLISH_API_KEY` in the agent if the store requires auth.

## Validation

In the agent repo, run `npm test` to validate skills. Checks include presence of `name` and `description`, valid frontmatter, and optional `metadata.sulala.requires.bins` parsing.
