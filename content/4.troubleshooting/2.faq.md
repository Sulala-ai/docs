## Do I need to run Integrations?

No. The agent runs fine without it. You need the Integrations service only if you want the agent to use OAuth-backed tools (Gmail, GitHub, Slack, Notion, etc.). In that case run Integrations, set the agent’s Portal URL and API key, and connect providers in the dashboard.

## Where are OAuth tokens stored?

Only in the **Integrations** service (e.g. in the file at `INTEGRATIONS_DATA_PATH`). The agent never stores tokens. It stores a **connectionId** and calls the Integrations API to get an access token when running a tool that needs that provider.

## Can I self-host the Store?

Yes. Deploy the Store app (e.g. Vercel or your own server). Set `NEXT_PUBLIC_STORE_URL` to your public URL. Point the agent at your store by setting `SKILLS_REGISTRY_URL` to `https://your-store.example.com/api/sulalahub/registry`. For persistent admin/submissions on Vercel you need sync (e.g. GitHub Action) or a self-hosted instance that can write to disk or a database.

## What’s the difference between a skill and a plugin?

- **Skill** — A Markdown doc (with optional frontmatter) that describes when and how to use tools. Loaded from user dir, workspace, managed dir, bundled context, or the store. No code; the agent has generic tools and follows the skill’s instructions.
- **Plugin** — Code in the agent’s `plugins/` directory that can implement lifecycle hooks (e.g. `onAgentBeforePromptBuild`, `onAgentAfterToolCall`). Used to extend or customize agent behaviour; not the same as a skill.

## Why don’t I see any skills in the registry?

- Ensure **`SKILLS_REGISTRY_URL`** is set in the agent to a valid registry endpoint (e.g. `https://hub.sulala.ai/api/sulalahub/registry` or your store’s `/api/sulalahub/registry`).
- Ensure the agent can reach that URL (no firewall blocking outbound HTTPS).
- If you run the store locally, use `http://localhost:3000/api/sulalahub/registry` (or the port you use) and that the store is running.

## How do I add my own skill without publishing to the store?

**Option 1 — Ask the AI chat to create one**  
In the dashboard chat, ask the agent to create a skill. For example: *"Create a hello world skill"* or *"Create an example skill called helloworld that says hello to the user."* The agent will create the skill folder with `README.md` and an empty `tools.yaml` under your workspace skills directory (e.g. `~/.sulala/workspace/skills/my/helloworld/`). Refresh skills or restart the gateway to load it.

**Option 2 — Add it manually**  
Put the skill under a directory the agent loads from. Recommended: **`~/.sulala/workspace/skills/<name>/README.md`** (or `SKILL.md`). Use a folder per skill with `name` and `description` in the Markdown frontmatter. Restart the gateway or rely on the skills watcher (`SKILLS_WATCH=1`) to pick it up. See [Skills authoring](/platform/skills-authoring).

## The agent says “invalid token” for an integration.

- In the dashboard **Settings → Portal** (or Integrations), confirm **Portal URL** and **API key** match the Integrations service.
- Ensure the **connection** for that provider was completed (OAuth finished and redirect back to the app). Reconnect the provider if needed.
- Ensure the skill or tool config references the correct **connectionId** for that provider.

## Is the agent exposed to the internet?

By default, no. The gateway binds to **127.0.0.1** (localhost only). Only processes on your machine can reach it unless you change `HOST` or put the agent behind a reverse proxy and explicitly expose it.
