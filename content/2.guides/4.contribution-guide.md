## Overview

Contributions are welcome to the **Agent**, **Integrations**, **Store**, and **documentation**. This guide covers running from source, where to make changes, code quality, and how to submit contributions.

## Running from source

### Agent

```bash
cd agent
npm install
cp .env.example .env
npm run dev
```

- **Gateway + watcher + scheduler:** `npm start` or `npm run dev` (watch mode).
- **Dashboard (dev):** `npm run dashboard` — UI on its own port; set `VITE_GATEWAY_URL=http://127.0.0.1:2026` in `dashboard/.env`.
- **Tests:** `npm test` (Vitest). **Lint:** `npm run lint` (ESLint on `src/`).
- **CLI:** `npm run cli -- <args>` (e.g. `npm run cli -- status`, `npm run cli -- skill list`).

### Integrations

```bash
cd integrations
npm install
cp .env.example .env
npm run dev
```

Default: http://127.0.0.1:1717. See [Integrations](/platform/integrations) for env vars (port, callback URL, provider client IDs/secrets).

### Store

```bash
cd store
pnpm install
cp .env.example .env
pnpm dev
```

Default: http://localhost:3000. Set `NEXT_PUBLIC_STORE_URL` for production builds. **Lint:** `pnpm lint`. **Sync registry:** `pnpm sync-registry` (from repo with agent registry).

### Documentation (this site)

```bash
cd docs
npm install
npm run dev
```

Edit Markdown under `docs/content/`. The site uses Docus/Nuxt Content; structure is folder-based (e.g. `1.getting-started/`, `2.guides/`, `3.platform/`, `4.troubleshooting/`).

## Where to make changes

| Area | Main paths | Notes |
|------|------------|-------|
| **Agent** | `agent/src/` — `gateway/`, `watcher/`, `scheduler/`, `ai/`, `agent/`, `db/`, `plugins/` | TypeScript; entry is `src/index.ts`. Dashboard: `agent/dashboard/` (React + Vite). |
| **Integrations** | `integrations/src/` — `providers/`, server and OAuth callback | Add a provider in `src/providers/<name>.ts` and register in `src/providers/index.ts`. |
| **Store** | `store/app/`, `store/features/`, `store/components/` | Next.js app. Data: `store/data/` (registry, skills, store-meta). |
| **Docs** | `docs/content/` | Markdown with optional frontmatter and Docus components. |

## Code quality

- **Agent:** Run `npm run lint` and `npm test` before submitting. Fix lint errors; ensure tests pass.
- **Store:** Run `pnpm lint`. Fix any reported issues.
- **Integrations:** No test script in the example; add tests if you introduce new behaviour. Keep provider code consistent with existing providers.
- **Docs:** Use clear headings, consistent linking (`/platform/agent`-style paths), and the same frontmatter pattern (`title`, `description`, `navigation.icon`) as existing pages.

## Submitting changes

1. **Fork** the repository (if external) or create a **branch** from the default branch.
2. **Make your changes** in a focused branch (e.g. `fix/gateway-auth`, `feat/integrations-slack`, `docs/contribution-guide`).
3. **Run lint and tests** for the areas you changed (agent, store, etc.).
4. **Commit** with clear messages (e.g. "fix: gateway API key check", "docs: add contribution guide").
5. **Open a Pull Request** with a short description of what changed and why. Link any related issues.

If the project uses a specific workflow (e.g. conventional commits, required approvals), follow the repo’s README or maintainer instructions.

## Contributing skills

- **Publish via the Store:** Use the store’s **Submit** flow or the agent dashboard **Publish to store**. See [Skills authoring](/platform/skills-authoring) for format (README.md + optional tools.yaml).
- **Contribute to the hub’s catalog:** If you maintain the registry or store data, add skills under `store/data/skills/<slug>/` (README.md + optional tools.yaml) and an entry in `store/data/registry.json`. Run `pnpm sync-registry` when syncing from another repo.

## Contributing integrations

- **New OAuth provider:** Add a new file under `integrations/src/providers/` (e.g. `slack.ts`) with auth URL and token exchange. Register it in `integrations/src/providers/index.ts`. Use env vars for client ID/secret (e.g. `SLACK_CLIENT_ID`, `SLACK_CLIENT_SECRET`). See [Integrations](/platform/integrations) and `integrations/INTEGRATIONS_ROADMAP.md`.
- **New agent tools for an existing provider:** Add or extend tools in the agent (e.g. `agent/src/agent/` or provider-specific modules) that use `getConnectionToken(connectionId)` or the integrations API. Document in the Integrations roadmap if relevant.

## Contributing documentation

- **Fix or extend existing pages:** Edit the relevant file under `docs/content/`. Use the same structure (sections, tables, code blocks) as nearby pages.
- **New page:** Add a new Markdown file in the right section (e.g. `2.guides/5.my-guide.md`). Set frontmatter: `title`, `description`, and `navigation.icon` (e.g. `i-lucide-book-open`). If the section has a `.navigation.yml`, the new page will appear in the sidebar.
- **Index:** If the new page should be linked from the docs home, add a card or link in `docs/content/index.md` under the appropriate section.

## Questions

If the repo has a GitHub Discussions, issue tracker, or community channel, use those for questions. For bugs or feature requests, open an issue with a clear description and steps to reproduce (for bugs).
