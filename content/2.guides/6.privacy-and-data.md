## Overview

The Sulala Platform is designed so that sensitive data and credentials stay under your control. This page summarizes what runs locally, what is sent where, and how to reason about privacy.

## What runs locally

### Agent

- **Gateway, file watcher, task scheduler, AI orchestration, plugins** — All run on your machine (default `127.0.0.1`). No Sulala server receives your file paths, task payloads, or chat content unless you configure webhooks or remote services.
- **SQLite database** — Tasks, logs, file state, and AI result cache are stored in a local DB (e.g. `./data/sulala.db` or path set by `DB_PATH`). Never sent off-device by the agent itself.
- **Skill content** — Loaded from local directories (`~/.sulala/workspace/skills/`, etc.) or from the registry URL you set. When you install from a hub, the agent fetches skill Markdown and tools.yaml from that URL; the hub (e.g. hub.sulala.ai or your store) sees the request.
- **Ollama** — If you use Ollama as the AI provider, models run locally. No API key; no data sent to a third-party AI provider.

### Integrations service

- **OAuth tokens** — Stored in a file (e.g. `~/.sulala/integrations.json`) or in your Supabase project if you set `INTEGRATIONS_STORE=database`. Tokens are not sent to Sulala; they stay in the storage you configure.
- **OAuth flow** — When you connect a provider (GitHub, Gmail, etc.), your browser talks to the provider and then to your Integrations service. The provider may log the auth request per its own policy; the Integrations service only stores the resulting tokens locally (or in your DB).

### Store

- **Registry and skill content** — When the agent uses `SKILLS_REGISTRY_URL`, it fetches the registry and skill Markdown (and optionally tools.yaml) from that host. If the host is hub.sulala.ai or your own store, that server sees the GET requests (URL, IP). Request bodies are not used for registry/skill reads.
- **Submissions** — If you submit a skill (via the store form or “Publish to store” from the dashboard), the skill content and metadata you send go to that store’s backend (and may be stored in files or a database you control if you self-host).

## What is sent to external services

- **AI providers (OpenAI, Claude, Gemini, OpenRouter)** — When you use a non-local provider, the agent sends prompts (system prompt, message history, tool calls) and receives replies over the provider’s API. Their privacy and data policies apply. Use Ollama for fully local inference if you want no AI data to leave your machine.
- **Portal / Integrations API** — When the agent runs a tool that needs an OAuth connection, it calls your Integrations service (or Portal) with a `connectionId` to get a short-lived access token. The token request is between your agent and your Integrations instance (or the Portal you configured); the actual API calls (e.g. to Gmail, GitHub) are made by the agent using that token and go to those providers.
- **Store / hub** — Registry and skill fetches (GET) go to the host in `SKILLS_REGISTRY_URL`. Submissions (POST) go to the store’s submission endpoint if you use “Publish to store” or the web form.
- **Webhooks** — If you set `WEBHOOK_URL` or `WEBHOOK_URLS`, the agent may POST events (e.g. file_event, task_done) to those URLs. Only you control those endpoints.

## Credentials

- **Agent** — API keys (AI, optional gateway key) are read from `.env` on the machine where the agent runs. Not sent to Sulala; only to the services you configure (e.g. OpenAI, Portal).
- **Integrations** — OAuth client IDs and secrets live in the Integrations service `.env`. Tokens are stored in the Integrations store (file or DB). The agent never stores OAuth tokens; it only stores connection IDs and fetches tokens when needed.
- **Store** — Clerk, Supabase, Stripe, and admin secret are in the store’s `.env`. Never commit `.env` files.

## Summary

| Data | Where it lives / goes |
|------|------------------------|
| Chat history, task queue, file paths | Local (agent DB and process). |
| Ollama usage | Local (no third-party AI). |
| Other AI providers | Prompts and replies go to that provider’s API. |
| OAuth tokens | Integrations store (your file or DB). Agent only holds connectionIds. |
| Registry/skill fetches | To the host in SKILLS_REGISTRY_URL (e.g. hub or your store). |
| Skill submissions | To the store backend you submit to. |
| Webhooks | To the URLs you set in the agent. |

For strictest privacy: use Ollama only, self-host Integrations and Store, and do not set webhooks or third-party AI keys.
