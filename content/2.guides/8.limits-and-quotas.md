## Overview

This page summarizes configurable limits and quotas so you can tune behaviour and understand why a request might be throttled or truncated.

## Agent

### Rate limiting

| Variable | Default | Description |
|----------|---------|-------------|
| `RATE_LIMIT_MAX` | 100 | Max number of requests allowed per window. |
| `RATE_LIMIT_WINDOW_MS` | 60000 | Window length in milliseconds (e.g. 60_000 = 1 minute). |

When exceeded, the gateway may respond with 429 or similar. Adjust in `.env` if you need higher or lower throughput.

### Agent run and context

| Variable | Default | Description |
|----------|---------|-------------|
| `AGENT_TIMEOUT_MS` | 300000 | Max duration (ms) for a single agent turn. 0 = no timeout. Request body can override with `timeout_ms`. |
| `AGENT_MAX_CONTEXT_TOKENS` | 70000 | Approximate token cap for the conversation sent to the model. Older messages are dropped until under the cap. |
| `AGENT_MAX_HISTORY_MESSAGES` | — | If set, max number of messages kept in context; older messages are dropped. |

Larger context allows longer conversations but increases latency and cost for API-based models.

### Tools

- **AGENT_TOOL_ALLOWLIST** — If set, only the listed tool names are available. No per-tool rate limit in the agent; external APIs (e.g. OpenAI, Stripe) have their own quotas.
- **AGENT_TOOL_RETRY_COUNT** — Number of retries for failed tool calls (default 2). Not a quota; affects resilience.

## Integrations

- No built-in rate limit in the Integrations service itself. OAuth providers (GitHub, Google, etc.) enforce their own limits on auth and API calls.
- When using **Portal**, the Portal or gateway may apply additional rate limits; see Portal documentation if applicable.

## Store

- **Registry and skill content** — Served by your Store app (e.g. Next.js). Limits depend on your hosting (Vercel, server, etc.) and any rate limiting you add.
- **Submissions** — Typically limited by store implementation (e.g. one submission per slug per user, or moderation queue size). No global doc-wide quota; check the store app or admin docs.
- **Paid skills** — Stripe enforces its own limits (e.g. API rate limits, Connect account limits). See [Stripe documentation](https://stripe.com/docs).

## AI providers (external)

When the agent uses OpenAI, Claude, Gemini, or OpenRouter:

- **Token limits** — Each provider has max context and output tokens per request. The agent’s `AGENT_MAX_CONTEXT_TOKENS` and request `max_tokens` should stay within the provider’s limits.
- **Rate limits** — Requests per minute (RPM) and tokens per minute (TPM) depend on your API key and plan. Throttling or errors from the provider will surface as failed or delayed agent responses.

Check each provider’s documentation for current limits and pricing.

## Summary

| Area | What’s limited | Where to configure |
|------|----------------|--------------------|
| Agent | Requests per time window | `RATE_LIMIT_MAX`, `RATE_LIMIT_WINDOW_MS` |
| Agent | Turn duration | `AGENT_TIMEOUT_MS` |
| Agent | Context size | `AGENT_MAX_CONTEXT_TOKENS`, `AGENT_MAX_HISTORY_MESSAGES` |
| Integrations | (None in app) | OAuth providers have their own limits |
| Store | (Hosting-dependent) | Your deployment and any custom rate limiting |
| AI providers | Tokens, RPM/TPM | Provider dashboard and API plan |
